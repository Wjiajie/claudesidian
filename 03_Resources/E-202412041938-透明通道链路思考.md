## Alpha 合成(alpha compositing)

Alpha 合成可以用下面的简化三层光反射折射模型解释，下图的 $$foreground_A$$和 $$foreground_B$$两个图层允许有任意透明度[0, 1]，而 $$background$$的透明度是 1。

Alpha 合成为多图层编辑情况下的透明信息导出提供了理论基础，在保持层级顺序不变的情况下，允许随意打乱前景图层的顺序做混合，而结果一致，更多关于alpha合成的介绍，见[透明通道相关理论与常见问题](https://bytedance.larkoffice.com/docx/MjypdR8C6onhCIxcBytcy9bqn3d)。

## Swing 透明通道处理现状

Effect Swing 框架对传入纹理的透明通道的处理，常见有以下的场景：

1. 在混合shader中 $$foreground_A$$和 $$foreground_B$$使用alpha合成的over模式
    
2. 道具包（特效/转场）如果对输入纹理要求预乘（如一些模糊效果不能应用在透明通道，否则会有黑边），会在shader中做预乘操作。
    
3. Swing SDK 的一些模块要求输入预乘纹理，如果是非预乘纹理则会做预乘转换。

在 NodeGraphV2中的alphaState是如何传递的，可以从sourceAlphaState和targetAlphaState两个角度看：sourceAlphaState代表当前节点传入的纹理的alphaState，targetAlphaState代表这个节点希望接收的纹理的alphaState，如果sourceAlphaState和targetAlphaState不匹配，纹理转换应该执行。经过节点的处理后，纹理将会被转换成targetAlphaState。

### **sourceAlphaState**

sourceAlphaState标记当前节点传入的纹理的alphaState，在NodeGraphV2中，节点用Node表示，而sourceAlphaState这个纹理的状态是定义在JobData中，Node和JobData是一一对应的关系，这个映射关系定义在SwingJobGraph中，为了方便化简结构，可以认为Node也是具备唯一的alphaState标记，注意，它仅代表纹理的状态，NodeGraphV2的视角中，alphaState仅在Node之间传递。

```C++
class Node final : public std::enable_shared_from_this<Node>
{
    ...
    std::vector<AmazingEngine::Segment*> mSegments;
    std::function<JobDataPtr(SwingJobGraph*, SwingTimeInUS, DeviceTexture)> mJob;
    ...
}

struct alignas(64) JobData 
{
    SwingTimeInUS timeStamp{0};
    DeviceTexture outputTex{nullptr};
    int alphaState{0};
    ErrorCode retVal{ErrorCode::FAILED};
    ...
};

class SwingJobGraph : public JobSet
{
    std::unordered_map<SwingTimeInUS, std::unordered_map<Node*, JobDataPtr>> mTsJobData = {};
}
```

### targetAlphaState

  targetAlphaState则是另外一个视角，NodeGraphV2中把所有的node的mJob提交给任务系统，而每个mJob执行完成后，会返回JobData，其中包含了处理后的纹理和对应的alphaState，targetAlphaState标记的是每个mJob执行后，预期JobData中的alphaState要修改成哪种状态。这个targetAlphaState一般记录在Segment，Scene，持有scene的部分entity树的controller（下文称为customModule）中。

其实 AMGScene中并没有记录targetAlphaState，而是在scene的回调函数_sceneEventCallback中会假设它的targetAlphaState为segment的targetAlphaState。

  Segment-->Scene-->customModule三个层级对 targetAlphaState的处理逻辑大致如下：

- segment
    
      segment拥有一个m_alphaState的成员变量，这个变量在初始化时有一个默认值，比如videoSegment的默认m_alphaState = None，如果切换了混合模式，则解析混合模式资源包的config.json的alphaState并赋值给m_alphaState。featureSegment在资源包加载时也会读取config.json的alphaState并赋值给m_alphaState，如果找不到该字段则认为m_alphaState = Premultiplied。
    
      m_alphaState代表这个片段经过(单scene或者多scene)渲染后，预期输出的纹理的alphaState。
    
    个人觉得在_sceneEventCallback的纹理转换中，不应该拿m_alphaState这个成员作为判断当前输入纹理是否应该转换的标记，因为在多scene的逻辑中，m_alphaState并不能代表当前渲染scene的预期输出alphaState。而是增加标记当前scene的targetAlphaState的方法：
    
    1. 在segment中增加管理的变量，比如 std::unordered_map<Scene*, AMGAlphaStateType> m_sceneTargetAlphaStateMap;
        
    2. 直接在AMGScene中增加targetAlphaState变量
        
    3. BuiltinObjectAPI中增加sourceAlphaState和targetAlphaState，其中sourceAlphaState是当前输入纹理的 alphaState，targetAlphaState可以交给segment来控制业务逻辑，在_sceneEventCallback执行时，segment检索当前scene启动的模块情况，给出预期输出的alphaState。
        
    
      videoSegment当前对alphaState的管理会有额外的逻辑，针对videoSegment的混合节点，videoSegment会用额外的m_inputTexAlphaState， m_backgroundAlphaState， m_blendAlphaState来标记的最终执行到混合节点时，设置给混合shader的fgTex, bgTex 和 blendTex的alphaState。
    
    - **m_inputTexAlphaState**
        
            m_inputTexAlphaState的状态变化比较多，是否加载局部特效，视频动画，蒙版都会影响m_inputTexAlphaState的值。
        
    - **m_backgroundAlphaState**
        
            m_backgroundAlphaState固定来自于画布纹理的alphaState
        
    - **m_blendAlphaState**
        
            由于混合节点目前固定在videoSegment的最后执行，可以认为m_blendAlphaState等同于videoSgement.m_alphaState，即videoSgement最终输出纹理的alphaState。
        
        m_blendAlphaState可以由 m_inputTexAlphaState以及m_backgroundAlphaState联合决定，或者当设置了混合模式时，由混合模式的道具包配置决定， m_blendAlphaState最终的值应该秉承"最小转换"的原则，能避免纹理的转换，宁可修改m_blendAlphaState的值，也不要做纹理转换。
        
        (只是猜测，后面可以验证下)
        
- Scene & customModule
    
      一般情况下，Scene的预期输出alphaState和它归属的sgement的预期输出的alphaState是一致的，比如特效、贴纸、文字、文字模板等segment，从历史的实现上，一般设计的shader都是预乘的写法，只有特定的资源包会根据需求实现为非预乘（比如调节资源包），这些情况下，直接读取segment.m_alphaState作为Scene预期输出alphaState即可。但是会有特殊情况，比如 videoSegment在加载的混合模式、视频动画等资源，或者视频混合等模块（customModule），并不是以scene的形式存在，而是会有一个controller持有某个scene的部分entity树，这些模块其实也对输入纹理是否预乘有要求（即这些模块也会有一个自己的targetAlphaState），此时scene的targetAlphaState不能简单取自segment.m_alphaState，而是需要根据scene内这些customModule的激活情况，给出scene的targetAlphaState。
