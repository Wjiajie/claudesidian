这篇文章核心围绕 “AI 智能体的有效上下文工程” 展开，提出其是提示工程的自然演进，核心目标是在 LLM 有限注意力预算下，筛选最小化的高信号 tokens 以实现预期行为，同时系统阐述了相关概念、重要性、实施方法及长周期任务解决方案。

### 一、核心概念：上下文工程 vs 提示工程

文章明确二者的演进关系与核心差异，强调上下文工程是应对智能体多轮交互需求的必然方向：

1. **提示工程**：侧重 “编写和组织 LLM 指令”，聚焦一次性任务（如分类、文本生成），核心是优化系统提示等静态指令。
2. **上下文工程**：侧重 “迭代管理 LLM 推理过程中的全部 token 集合”，涵盖系统指令、工具、外部数据、对话历史等动态信息；因智能体需多轮循环交互，需从不断变化的信息池中筛选出有限上下文窗口所需内容，是持续的 curated 过程。

### 二、上下文工程的重要性：LLM 的 “注意力局限”

文章指出，LLM 的固有特性决定了上下文必须被视为 “有限资源”：

1. **上下文衰减（Context Rot）**：随着上下文窗口 token 数量增加，LLM 准确召回信息的能力会下降，所有模型均存在此问题（仅衰减程度不同）。
2. **注意力预算有限**：类似人类工作记忆，LLM 的注意力是有限资源，新增 token 会消耗预算；且 transformer 架构导致 token 间 pairwise 关系呈 n² 复杂度，长序列下注意力分散。
3. **长序列适应性弱**：训练数据中短序列更常见，模型对长序列上下文依赖的处理能力较弱，即便通过位置编码插值适配长序列，信息检索和长程推理精度仍会下降。

### 三、有效上下文的构成原则

文章提出，构建有效上下文需围绕 “高信号、轻量化” 展开，具体覆盖三大核心组件：

1. **系统提示**：需清晰且适度，避免两个极端 —— 不硬编码复杂逻辑（易脆弱、难维护），也不提供模糊指导（缺乏具体信号）；建议用结构化格式（如 XML 标签、Markdown 标题）拆分模块（背景、指令、工具指南等），且以 “最小必要信息” 为目标，先测试基础提示再按需补充。
2. **工具设计**：工具是智能体与环境交互的桥梁，需满足 “高效性”—— 功能独立无重叠、鲁棒抗错、用途明确；避免工具集臃肿（若人类无法判断工具用途，LLM 更无法判断），参数需描述清晰以契合 LLM 优势。
3. **示例（Few-shot Prompting）**：需精选 “多样、典型” 的案例，传递预期行为，避免堆砌边缘案例；对 LLM 而言，优质示例的信息密度远高于文字规则。

### 四、上下文检索与智能体搜索：从 “预检索” 到 “即时检索”

文章指出当前智能体上下文设计的趋势是 “动态适配”，核心策略包括：

1. **即时（Just in Time）检索**：不预先处理所有数据，而是通过轻量标识（文件路径、网页链接等），在推理时用工具动态加载所需数据；例如 Claude Code 通过编写定向查询、使用 Bash 命令（head/tail）分析大数据，无需加载完整数据。
2. **混合检索策略**：结合 “预检索”（提升速度）与 “即时检索”（保证准确性），根据任务灵活调整；例如 Claude Code 会预先将 CLAUDE.md 放入上下文，同时用 glob/grep 工具即时检索其他文件，规避索引过期问题。
3. **渐进式披露**：智能体通过自主探索逐步获取上下文（如文件名暗示用途、时间戳提示相关性），仅保留必要信息在工作记忆，用笔记补充持久化，避免信息过载。

### 五、长周期任务的上下文管理技术

针对 token 超窗、上下文污染问题，文章提出三大核心技术：

1. **压缩（Compaction）**：当上下文接近窗口上限时，总结关键内容（如架构决策、未解决 bug），丢弃冗余信息（如旧工具结果），用总结开启新上下文；例如 Claude Code 会保留最近访问的 5 个文件 + 压缩后的历史，保证任务连续性。
2. **结构化笔记（Agentic Memory）**：智能体将关键信息（进度、目标、策略）存储到上下文外的外部记忆（如 NOTES.md），需用时再加载；例如 Claude 玩 Pokémon 时，通过笔记跟踪训练步数、地图探索情况、战斗策略，实现多小时任务的连贯性。
3. **子智能体架构**：主智能体负责整体规划，子智能体专注专项任务（如深度检索、技术分析），子智能体可使用大量 token 但仅返回精简结果（1000-2000 token）；该模式在复杂研究任务中表现优于单智能体，实现 “关注点分离”。

### 六、结论

1. 上下文工程标志着 LLM 应用从 “优化指令” 向 “优化信息筛选” 的转变，核心原则始终是 “找到最小化的高信号 tokens 以实现预期行为”。
2. 随着模型能力提升，智能体自主性会增强，对预设工程的依赖降低，但 “上下文是有限资源” 的本质不变，精心管理仍是构建可靠智能体的关键。
3. 文章建议开发者可通过 Claude 开发者平台实践相关技术，并参考其内存与上下文管理指南。

[Anthropic的工程团队](https://www.anthropic.com/engineering)

在应用人工智能领域，提示词工程成为关注焦点几年后，一个新术语开始崭露头角： **上下文工程** 。使用语言模型进行构建，正从寻找合适的提示词和短语，更多地转向回答一个更广泛的问题：“什么样的上下文配置最有可能让我们的模型产生期望的行为？”

**上下文** 指的是从大型语言模型（LLM）中采样时包含的一组标记。当前面临的 **工程** 问题是，在LLM固有的约束条件下优化这些标记的效用，以持续达成预期结果。有效地驾驭LLM往往需要 *结合上下文思考* ，换句话说：要考虑LLM在任何特定时间所能获取的整体状态，以及该状态可能产生的潜在行为。

在这篇文章中，我们将探讨新兴的语境工程艺术，并提供一个经过优化的心智模型，用于构建可操控、高效的智能体。

## 上下文工程与提示词工程

在Anthropic，我们将上下文工程视为提示词工程的自然发展。提示词工程指的是为获得最佳结果而编写和组织大语言模型指令的方法（详见 [我们的文档](https://docs.anthropic.com/en/docs/build-with-claude/prompt-engineering/overview) ，其中概述了实用的提示词工程策略）。 **上下文工程** 指的是在大语言模型推理过程中，筛选和维护最佳令牌（信息）集的一系列策略，其中包括提示词之外可能出现在该过程中的所有其他信息。

在大语言模型工程应用的早期，提示词编写是人工智能工程工作的最大组成部分，因为除日常聊天交互之外的大多数用例都需要针对一次性分类或文本生成任务优化的提示词。顾名思义，提示词工程的主要重点在于如何编写有效的提示词，尤其是系统提示词。然而，随着我们着手开发更强大的智能体——这些智能体能够进行多轮推理并在更长的时间范围内运行，我们需要相应的策略来管理整个上下文状态（系统指令、工具、 [模型上下文协议](https://modelcontextprotocol.io/docs/getting-started/intro) （MCP）、外部数据、消息历史等）。

一个循环运行的智能体生成越来越多的数据，这些数据可能与下一轮推理相关，而这些信息必须进行周期性的优化。上下文工程是一门艺术和科学，它负责从不断演变的海量潜在信息中筛选出适合放入有限上下文窗口的内容。

![Prompt engineering vs. context engineering](https://www.anthropic.com/_next/image?url=https%3A%2F%2Fwww-cdn.anthropic.com%2Fimages%2F4zrzovbb%2Fwebsite%2Ffaa261102e46c7f090a2402a49000ffae18c5dd6-2292x1290.png&w=3840&q=75)

与编写提示词这一离散任务不同，上下文工程是一个迭代的过程，并且每次我们决定向模型传递什么内容时，筛选阶段就会发生。

## 为什么上下文工程对构建高性能智能体至关重要

尽管大型语言模型（LLMs）速度快，且能够处理越来越大的数据量，但我们观察到，它们和人类一样，在某个节点会注意力不集中或陷入困惑。针对“大海捞针”式基准测试的研究揭示了“语境衰减”这一概念：随着语境窗口中标记数量的增加，模型从该语境中准确回忆信息的能力会下降。

虽然有些模型的性能下降更为平缓，但这一特征在所有模型中都存在。因此，上下文必须被视为一种边际收益递减的有限资源。与人类一样，大语言模型（LLMs）也有“注意力预算”，在解析大量上下文时会用到这一预算，而人类则有 [有限的工作记忆容量](https://journals.sagepub.com/doi/abs/10.1177/0963721409359277) 。每引入一个新标记都会在一定程度上消耗这一预算，这就使得我们更有必要精心筛选可供大语言模型使用的标记。

这种注意力稀缺源于大型语言模型的架构限制。大型语言模型基于 [Transformer架构](https://arxiv.org/abs/1706.03762) ，这使得每个标记能够对整个上下文中的 [其他每个标记进行注意力分配](https://huggingface.co/blog/Esmail-AGumaan/attention-is-all-you-need) 。这导致n个标记产生n²个成对关系。

随着上下文长度的增加，模型捕捉这些成对关系的能力会被稀释，这就在上下文大小和注意力集中之间造成了一种自然的张力。此外，模型的注意力模式是从训练数据分布中形成的，在这些数据中，较短的序列通常比长序列更常见。这意味着模型在处理全上下文依赖关系方面经验更少，专门的参数也更少。

诸如 [位置编码插值](https://arxiv.org/pdf/2306.15595) 之类的技术，能够通过将模型适配到最初训练时的较小上下文，从而让模型处理更长的序列，不过这会在一定程度上降低对标记位置的理解能力。这些因素会形成性能梯度，而非陡峭的性能悬崖：模型在更长的上下文中仍然保有较强的能力，但与在较短上下文中的表现相比，在信息检索和长程推理方面的精度可能会有所下降。

这些现实情况意味着，深思熟虑的上下文工程对于构建高性能智能体至关重要。

## 有效的上下文结构

鉴于大型语言模型受限于有限的注意力预算， *良好的* 上下文工程意味着找到 *最小的* *可能的* 高信号标记集，以最大化某些预期结果的可能性。实施这种做法说起来容易做起来难，但在接下来的部分中，我们将概述这一指导原则在上下文的不同组成部分中的实际意义。

**系统提示词** 应当极其清晰，使用简单直接的语言，以适合智能体的 *恰当高度* 呈现观点。恰当高度是介于两种常见失效模式之间的“ Goldilocks 区域”（最佳区域）。一种极端情况是，工程师在提示词中硬编码复杂、脆弱的逻辑，以获取精确的智能体行为。这种方法会造成脆弱性，并随着时间的推移增加维护复杂度。另一种极端情况是，工程师有时会提供模糊、高层级的指导，无法为大语言模型提供关于期望输出的具体信号，或者错误地假设存在共享语境。最佳高度需要达到一种平衡：足够具体以有效引导行为，同时又足够灵活，为模型提供强大的启发式方法来指导行为。

我们建议将提示词组织成不同的部分（例如 `<background_information>` 、 `<instructions>` 、 `## Tool guidance` 、 `## Output description` 等），并使用XML标签或Markdown标题等技术来划分这些部分，不过随着模型能力的提升，提示词的确切格式可能变得不那么重要了。

无论你决定如何构建系统提示词，都应努力用最少的信息完整勾勒出你期望的行为。（请注意，“最少”并不一定意味着“简短”；你仍然需要预先向智能体提供足够的信息，以确保它遵循预期的行为。）最好的做法是，先使用可用的最佳模型测试一个精简的提示词，看看它在你的任务中的表现如何，然后根据初始测试中发现的失效模式，添加清晰的指令和示例来改进性能。

**工具** 使智能体能够与其环境交互，并在工作时获取新的、额外的上下文。由于工具定义了智能体与其信息/行动空间之间的约定，因此工具促进效率至关重要，这既包括返回令牌高效的信息，也包括鼓励高效的智能体行为。

在《为AI智能体编写工具——借助AI智能体》中，我们探讨了如何构建能被大语言模型充分理解且功能重叠极少的工具。与设计良好的代码库的功能类似，工具应具备独立性、强大的容错能力，并且其预期用途要极其明确。输入参数同样应具有描述性、明确性，并能发挥模型固有的优势。

我们发现最常见的失败模式之一是工具集过于臃肿，涵盖了过多功能，或者导致在选择使用哪种工具时出现模糊的决策点。如果人类工程师都无法明确说出在特定情况下应该使用哪种工具，那么就不能指望AI智能体做得更好。正如我们稍后将讨论的，为智能体精心挑选一套最小可行的工具集，还能在长期交互中实现更可靠的维护和上下文精简。

提供示例，也就是所谓的少样本提示，是一种广为人知的最佳实践，我们仍然强烈建议采用这种方法。然而，团队往往会在提示中堆砌一长串边缘案例，试图阐明大语言模型在特定任务中应遵循的每一条可能的规则。我们不建议这样做。相反，我们建议精心挑选一组多样化的、标准的示例，这些示例能够有效地展现智能体的预期行为。对于大语言模型而言，示例就是那些“胜过千言万语的图画”。

我们针对上下文不同组成部分（系统提示 **、** 工具 **、** 示例 **、** 消息历史等）的总体指导原则是，要用心设计，确保上下文内容丰富且简洁。现在，让我们深入探讨如何在运行时动态检索上下文。

## 上下文检索与智能体搜索

在《构建高效的AI智能体》中，我们强调了基于大语言模型（LLM）的工作流与智能体之间的区别。自那篇文章发表以来，我们逐渐倾向于给智能体下一个简单的定义：大语言模型自主地在一个循环中使用工具。

在与客户的合作过程中，我们发现该领域正逐渐聚焦于这一简单范式。随着底层模型的能力不断增强，智能体的自主性水平也能随之提升：更智能的模型使智能体能够独立应对复杂的问题场景，并从错误中恢复。

我们现在看到工程师在思考如何为智能体设计上下文方面发生了转变。如今，许多原生人工智能应用程序都采用某种形式的基于嵌入的推理前检索，以呈现重要的上下文供智能体进行推理。随着该领域向更具智能体特性的方法转变，我们越来越多地看到各团队通过“即时”上下文策略来增强这些检索系统。

与预先处理所有相关数据不同，采用“即时”方法构建的智能体保留轻量级标识符（文件路径、存储的查询、网页链接等），并使用这些引用在运行时通过工具将数据动态加载到上下文中。Anthropic的智能编码解决方案 [Claude Code](https://www.anthropic.com/claude-code) 就采用这种方法对大型数据库进行复杂的数据分析。该模型可以编写有针对性的查询、存储结果，并利用像head和tail这样的Bash命令来分析大量数据，而无需将完整的数据对象加载到上下文中。这种方法类似于人类的认知方式：我们通常不会记住全部的信息 corpus，而是借助文件系统、收件箱和书签等外部组织和索引系统，按需检索相关信息。

除了存储效率之外，这些引用的元数据还提供了一种有效优化行为的机制，无论是明确提供的还是凭直觉获得的。对于在文件系统中运行的智能体而言，名为 `test_utils.py` 的文件出现在 `tests` 文件夹中，其用途与位于 `src/core_logic/` 中的同名文件不同。文件夹层次结构、命名约定和时间戳都提供了重要信号，帮助人类和智能体理解如何以及何时利用信息。

让智能体自主导航和检索数据还能实现渐进式披露——换句话说，就是允许智能体通过探索逐步发现相关语境。每一次交互都会产生为下一个决策提供依据的语境：文件大小暗示着复杂度；命名规则透露出用途；时间戳可以作为相关性的替代指标。智能体能够层层构建理解，在工作记忆中只保留必要的信息，并借助记笔记的策略来增强信息的持久性。这种自我管理的语境窗口能让智能体专注于相关的信息子集，而不会淹没在大量但可能无关的信息中。

当然，这其中存在一种权衡：运行时探索比检索预先计算的数据要慢。不仅如此，还需要有主见且深思熟虑的工程设计，以确保大语言模型拥有合适的工具和启发式方法，从而有效地在其信息环境中导航。如果没有适当的指导，智能体可能会因误用工具、陷入死胡同或无法识别关键信息而浪费上下文。

在某些场景下，最有效的智能体可能会采用混合策略，预先检索部分数据以提高速度，并自主决定是否进行进一步的自主探索。“适当”自主程度的决策界限取决于具体任务。Claude Code 就是一个采用这种混合模型的智能体： [CLAUDE.md](http://claude.md/) 文件会被直接简单地放入上下文，而像 glob 和 grep 这样的基本工具使其能够在环境中导航并实时检索文件，从而有效规避了索引过时和语法树复杂的问题。

混合策略可能更适合内容动态性较低的场景，例如法律或金融工作。随着模型能力的提升，智能体设计将趋向于让智能模型自主智能地行动，人类的整理工作会逐渐减少。鉴于该领域的快速发展，“做最简单且有效的事”很可能仍是我们对基于Claude构建智能体的团队的最佳建议。

### 长程任务的上下文工程

长时任务要求智能体在动作序列中保持连贯性、上下文一致性和目标导向行为，而这些动作序列的token数量会超过大语言模型的上下文窗口。对于那些需要数十分钟到数小时连续工作的任务，例如大型代码库迁移或综合性研究项目，智能体需要专门的技术来应对上下文窗口大小的限制。

等待更大的上下文窗口似乎是一种显而易见的策略。但在可预见的未来，各种规模的上下文窗口很可能都会受到上下文污染和信息相关性问题的影响——至少在需要智能体发挥最佳性能的情况下是如此。为了让智能体能够在更长的时间范围内有效工作，我们开发了一些直接解决这些上下文污染限制的技术：压缩、结构化笔记和多智能体架构。

**压缩**

压缩是指当对话接近上下文窗口限制时，对其内容进行总结，并利用该总结重新开启一个新的上下文窗口。压缩通常是上下文工程中提升长期连贯性的首要手段。其核心在于以高保真的方式提炼上下文窗口的内容，使智能体能够在性能下降最小的情况下继续运行。

例如，在Claude Code中，我们通过将消息历史传递给模型来实现这一点，以总结和压缩最关键的细节。该模型会保留架构决策、未解决的漏洞和实现细节，同时丢弃冗余的工具输出或消息。然后，智能体可以凭借这种压缩后的上下文以及最近访问的五个文件继续工作。用户无需担心上下文窗口的限制，就能获得连贯性。

压缩的艺术在于选择保留什么和舍弃什么，因为过度激进的压缩可能会导致一些微妙但关键的上下文信息丢失，而这些信息的重要性往往在之后才会显现出来。对于实施压缩系统的工程师，我们建议在复杂的智能体轨迹上仔细调整提示词。首先要最大限度地提高召回率，确保压缩提示词能从轨迹中捕捉到每一个相关信息，然后通过迭代来消除多余内容，提高精确度。

低价值冗余内容的一个例子是清理工具调用和结果——一旦某个工具在消息历史深处被调用过，智能体为何还需要再次查看原始结果呢？压缩处理中最安全、影响最小的方式之一就是清理工具结果，这也是最近在Claude开发者平台上推出的一项功能。

**结构化笔记法**

结构化笔记法，或称智能体记忆，是一种智能体定期将笔记写入上下文窗口之外的持久化记忆中的技术。这些笔记会在之后被重新拉入上下文窗口。

这一策略能以最小的开销提供持久化内存。就像Claude Code创建待办清单，或者您的自定义智能体维护一个NOTES.md文件一样，这种简单的模式使智能体能够跟踪复杂任务的进度，保留关键上下文和依赖关系——否则这些信息会在数十次工具调用中丢失。

[Claude玩《宝可梦》](https://www.twitch.tv/claudeplayspokemon) 展示了在非编码领域中，记忆如何改变智能体的能力。该智能体在数千个游戏步骤中保持精确记录——追踪各种目标，例如“在过去的1234步中，我一直在1号道路训练我的宝可梦，皮卡丘已朝着10级的目标提升了8级”。在没有任何关于记忆结构的提示的情况下，它会绘制已探索区域的地图，记住已解锁的关键成就，并留存关于战斗策略的战略性笔记，这些笔记有助于它了解哪些攻击对不同对手最有效。

在上下文重置后，智能体会阅读自己的笔记，并继续数小时的训练序列或地牢探索。这种跨总结步骤的连贯性，使得仅依靠大语言模型的上下文窗口来存储所有信息时无法实现的长周期策略成为可能。

作为我们 [Sonnet 4.5版本发布](https://www.anthropic.com/effective-context-engineering-for-ai-agents) 的一部分，我们在Claude开发者平台上推出了处于公开测试阶段的 [记忆工具](http://anthropic.com/news/context-management) ，该工具通过基于文件的系统，能更轻松地存储和查阅上下文窗口之外的信息。这使得智能体能够随着时间的推移建立知识库，在不同会话中保持项目状态，并且无需将所有内容都放在上下文中就能参考之前的工作。

**子智能体架构**

子智能体架构为突破上下文限制提供了另一种方法。不再由单个智能体尝试在整个项目中维持状态，而是由专门的子智能体在清晰的上下文窗口中处理针对性任务。主智能体凭借高层级计划进行协调，而子智能体则执行深入的技术性工作或使用工具查找相关信息。每个子智能体可能会进行广泛探索，使用数万个甚至更多的令牌，但仅返回其工作的精简、提炼后的摘要（通常为1000-2000个令牌）。

这种方法实现了关注点的明确分离——详细的搜索上下文保持在子智能体内独立存在，而主导智能体则专注于综合和分析结果。在《我们如何构建多智能体研究系统》中讨论的这种模式，在复杂研究任务上比单智能体系统有显著改进。

这些方法之间的选择取决于任务的特点。例如：

- 压缩技术能在需要大量往复交流的任务中维持对话流畅性；
- 记笔记在具有明确里程碑的迭代开发中表现出色；
- 多智能体架构能处理复杂的研究和分析工作，在这些工作中，并行探索能带来收益。

即便模型在不断改进，在长时间交互中保持连贯性的挑战仍将是构建更高效智能体的核心问题。

## 结论

上下文工程代表了我们使用大语言模型（LLMs）进行构建的一种根本性转变。随着模型能力的不断增强，挑战不再仅仅是精心设计完美的提示词，而是要深思熟虑地筛选在每一步进入模型有限注意力预算的信息。无论你是在为长期任务实施压缩处理、设计高效的令牌工具，还是让智能体能够实时探索其所处环境，核心原则始终不变：找到最小的高信号令牌集，以最大程度提高实现预期结果的可能性。

我们概述的这些技术将随着模型的改进而不断发展。我们已经发现，更智能的模型需要更少的规范性设计，这使得智能体能够更自主地运行。但即便能力不断增强，将上下文视为一种宝贵且有限的资源，仍将是构建可靠、高效智能体的核心所在。
